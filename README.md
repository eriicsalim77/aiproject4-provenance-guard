# Provenance Guard

Backend API that classifies whether a piece of text was likely written by a human or generated by AI. Returns a confidence score, a transparency label, and gives creators a way to appeal misclassifications.

Built for CodePath AI201, Project 4.

---

## What it does

Platforms hosting creative work (writing, music descriptions, blog posts) increasingly need a way to verify whether submissions are human-authored. Perfect AI detection isn't possible — but a system that's honest about uncertainty, gives creators a path to push back, and logs every decision is achievable.

Provenance Guard is a Flask API that:

1. Accepts text via `POST /submit`
2. Runs it through two independent detection signals
3. Combines them into a calibrated confidence score
4. Returns one of three transparency labels (likely AI / uncertain / likely human)
5. Lets creators appeal via `POST /appeal`
6. Logs every decision in a structured audit trail

---

## Architecture

```
                          POST /submit
                                │
                                ▼
        ┌─────────── Detection pipeline ────────────┐
        ▼                                            ▼
  Signal 1: LLM check                         Signal 2: Stylometry
  (Groq Llama 3.3)                            (5 weighted sub-scores)
        │                                            │
        └────────────────────┬───────────────────────┘
                             ▼
                  Combined = 0.6*LLM + 0.4*stylo
                             │
                             ▼
                  Map to label variant
                  >= 0.75 → likely_ai
                  0.40 - 0.74 → uncertain
                  < 0.40 → likely_human
                             │
                             ▼
                  Write to audit log
```

An appeal looks up the original decision by `content_id`, marks it `under_review`, logs the appeal alongside the original decision, and confirms.

## Detection signals

### Signal 1 — LLM classification (`signals.llm_signal`)

Sends the text to Groq's `llama-3.3-70b-versatile` and asks it to judge human-vs-AI holistically — pacing, voice, idiosyncrasy, opinion. The model returns structured JSON `{"score": 0.X, "reasoning": "..."}` where `0.0` = clearly human and `1.0` = clearly AI. Temperature is `0.2` for consistent scoring. If the response can't be parsed, the signal falls back to a neutral `0.5`.

### Signal 2 — Stylometric heuristics (`signals.stylometric_signal`)

Pure-Python statistical analysis, no external libraries. Five sub-scores are normalized to `0.0–1.0` (higher = more AI-like) and combined as a weighted average:

| Sub-score | Weight | Signal |
|---|---|---|
| AI buzzword density | 0.40 | Frequency of formal/abstract words LLMs overuse (transformative, paradigm, furthermore, deployment, …) |
| Average sentence length | 0.25 | AI trends long: > 18 words avg → high |
| Punctuation density | 0.20 | Conventional density (~0.04–0.06) reads AI-like |
| Type-token ratio | 0.10 | Lexical diversity |
| Sentence-length variance | 0.05 | Uniformity (down-weighted — AI need not be perfectly uniform) |

Text under 30 words returns a neutral `0.5` with a note, since stylometry is unreliable on short input.

### Why two signals

They capture genuinely different properties — one semantic, one structural. When they agree, confidence is high. When they disagree, the combined score lands in the wide "uncertain" band, which is the honest answer and triggers an appeal path.

---

## Confidence scoring & labels

The combined score is weighted **60/40** in favor of the LLM (richer semantic context):

```
combined = 0.6 * llm_score + 0.4 * stylometric_score
```

| Combined score | Attribution | Label |
|---|---|---|
| `>= 0.75` | `likely_ai` | ⚠️ **Likely AI-generated.** Our system found strong signals that this content was produced by an AI model. The creator can appeal this label if it's incorrect. |
| `0.40 – 0.74` | `uncertain` | 🟡 **Uncertain attribution.** Our system found mixed signals. This content has been flagged for human review and may be either AI-assisted or human-written with a formal style. The creator can provide context through an appeal. |
| `< 0.40` | `likely_human` | ✅ **Likely human-written.** Our system found strong signals that this content was produced by a human author. |

The "uncertain" band is intentionally wide: on a creative platform, false positives (flagging a human's work as AI) are worse than false negatives.

## Setup

Requires Python 3.10+ and a [Groq API key](https://console.groq.com/).

```bash
# 1. Clone and enter the repo
git clone https://github.com/eriicsalim77/aiproject4-provenance-guard.git
cd aiproject4-provenance-guard

# 2. Create and activate a virtual environment
python -m venv .venv
source .venv/Scripts/activate      # Windows (Git Bash)
# .venv\Scripts\activate           # Windows (Command Prompt / PowerShell)
# source .venv/bin/activate        # macOS / Linux

# 3. Install dependencies
pip install -r requirements.txt
```

Create a `.env` file in the repo root (it's gitignored — never commit it):

```
GROQ_API_KEY=your_key_here
```

### Run

```bash
python app.py
```

The server starts on `http://localhost:5000`.

## API reference

### `POST /submit`

Classify a piece of text. **Rate limited: 10 requests/minute, 100/day per IP.**

**Request**
```json
{
  "text": "...",
  "creator_id": "demo-user-1"
}
```

**Response**
```json
{
  "content_id": "9865fd53-be8c-4397-a930-596ea4ca507f",
  "attribution": "likely_ai",
  "confidence": 0.7789,
  "llm_score": 0.9,
  "stylometric_score": 0.5972,
  "label": "⚠️ **Likely AI-generated.** ...",
  "timestamp": "2026-06-29T03:03:00Z"
}
```

### `POST /appeal`

Appeal a classification. Looks up the original entry by `content_id`, marks it `under_review`, and logs the appeal. Not rate limited.

**Request**
```json
{
  "content_id": "1c6b6824-3be1-414e-96c6-a1e50bda05ff",
  "creator_reasoning": "I am a non-native English speaker writing in an academic register..."
}
```

**Response**
```json
{
  "status": "appeal_received",
  "content_id": "1c6b6824-3be1-414e-96c6-a1e50bda05ff",
  "review_status": "under_review"
}
```

Returns `404 {"error": "content_id not found"}` if the id doesn't exist.

### `GET /log`

Return all audit log entries. Not rate limited.

**Response:** `{"entries": [ ... ]}`

## Audit log structure

Entries are stored as a JSON list in `audit_log.json` (SQLite would be the production choice). Fields per entry:

| Field | Notes |
|---|---|
| `timestamp` | ISO 8601 |
| `content_id` | UUID |
| `creator_id` | from the submission |
| `attribution` | `likely_ai` \| `uncertain` \| `likely_human` |
| `confidence` | combined score, 0.0–1.0 |
| `llm_score` | Signal 1 output |
| `stylometric_score` | Signal 2 output |
| `status` | `classified` \| `under_review` \| `appeal_received` |
| `appeal_reasoning` | appeal entries only |

---

## Rate limiting

`/submit` is limited to **10 requests/minute and 100/day per IP** (Flask-Limiter, in-memory storage). A legitimate creator submits a few pieces per session; tighter ceilings catch automation and abuse. `/appeal` and `/log` are not limited.
